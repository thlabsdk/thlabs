# Sprint 2.1 — Nav Auth State Design

**Date:** 2026-05-17
**Status:** Approved

## Goal

Surface authenticated session state in the THLabs navigation without introducing dashboard, account, or social-UI aesthetics. Anonymous users see a `Login` link; authenticated users see their email and a `Logout` action. The nav should feel like terminal/session identity.

---

## Architecture

### Component split

```
components/layout/
  Nav.tsx            ← async server component; owns auth fetching
  NavLinks.tsx       ← "use client"; owns usePathname() active-link logic
  NavAuthSection.tsx ← server component; purely presentational, no auth logic
  LogoutButton.tsx   ← "use client"; moved from app/login/; unchanged logic
  Container.tsx      (unchanged)
  Footer.tsx         (unchanged)
```

### Nav.tsx (server component)

- Remove `"use client"` directive
- Becomes `async` — calls `createClient()` from `lib/supabase/server`
- Reads `user` once: `const { data: { user } } = await supabase.auth.getUser()`
- Passes `userEmail={user?.email ?? null}` to `<NavAuthSection>`
- Delegates link rendering to `<NavLinks>`
- Auth fetching lives here and nowhere else in the nav tree

### NavLinks.tsx (client component)

- Extracts the existing `usePathname()` + link-map logic from the current Nav
- Accepts `links` as a prop (or uses the same static array)
- No auth knowledge whatsoever

### NavAuthSection.tsx (server component)

- Props: `{ userEmail: string | null }`
- No hooks, no `createClient()`, no auth fetching, no abstractions for future account systems
- Purely presentational: receives `userEmail`, renders one of two states:
  - `null` → `<Link href="/login">Login</Link>`
  - email string → `<span>{userEmail}</span>` + `<LogoutButton />`

### LogoutButton.tsx (client component)

- Moved from `app/login/LogoutButton.tsx` to `components/layout/LogoutButton.tsx`
- Logic unchanged: `supabase.auth.signOut()` then `router.refresh()`
- `router.refresh()` causes Next.js to re-fetch the RSC tree including `Nav`, which re-reads the now-null session — nav updates in place with no full page reload
- `app/login/page.tsx` import path updated to new location

## UI / Styling

Nav layout (schematic):

```
THLabs        About  Projects  Contact        dev@thlabs.dk  Logout
THLabs        About  Projects  Contact        Login
```

Separation between nav links and auth section: **gap only** (e.g. `ml-8` or `gap-8` in the flex container). No border, no divider, no panel visual. A border would introduce dashboard/account-UI energy that conflicts with the THLabs aesthetic.

### Anonymous state

```
Login
```

- `<Link href="/login">`
- Style: `text-sm text-muted hover:text-foreground transition-colors duration-150`
- Same visual weight as nav links — no emphasis, no button treatment

### Authenticated state

```
dev@thlabs.dk  Logout
```

- Email: `text-xs font-mono text-muted` — smaller, mono, clearly secondary
- Logout: `<button>` styled `text-sm text-muted hover:text-foreground transition-colors duration-150`
- Rendered as a flex row with a small gap between email and logout action
- No separator character, no badge, no pill, no avatar, no dropdown

---

## Constraints

- No global auth context
- No client-side auth provider (Supabase, React Context, Zustand, etc.)
- No middleware
- No avatars, dropdowns, profile settings, badges, pills, notifications
- `NavAuthSection` must remain strictly presentational — no logic beyond conditional rendering on `userEmail`

---

## Verification

| Scenario | Expected |
|---|---|
| Anonymous user visits any page | `Login` link visible in nav |
| Authenticated user visits any page | Email + `Logout` visible in nav |
| User logs out via nav | Nav updates immediately (router refresh), shows `Login` |
| Hard refresh while authenticated | Correct authenticated state, no flicker |
| Hard refresh while anonymous | Correct anonymous state |
| Production build | No type errors, no missing imports |
